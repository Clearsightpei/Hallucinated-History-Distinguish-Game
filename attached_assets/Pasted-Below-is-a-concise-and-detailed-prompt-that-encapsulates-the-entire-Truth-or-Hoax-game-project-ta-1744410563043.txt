### Prompt for Coding AI

**Project: Truth or Hoax Historical Story Game**

**Objective**: Create a web-based game where users are presented with two versions of a historical story—one true (labeled T) and one fake (labeled H)—and must select the true version. Store user inputs in a database to calculate individual and overall accuracy. Use Python with Flask and SQLite, deploy to Heroku, and ensure no login system is required. The user is a beginner familiar with Python and VS Code.

**Requirements**:

1. **Game Functionality**:
   - Display two versions of a historical story side by side (or stacked on mobile), labeled T and H (randomly assigned to true/fake to avoid bias).
   - Users click a button to select the story they believe is true.
   - Provide immediate feedback: “Correct! This is the true story. [explanation]” or “Incorrect. The other story is true. [explanation]”.
   - Include a “New Story” button to load another story pair.
   - Show user accuracy (% correct choices) and overall accuracy (% correct per story) via buttons.

2. **Stories**:
   - Store at least three story pairs in the database, including:
     - **S1: Moon Landing 1969**
       - True (T): “NASA’s Apollo 11 landed humans on the moon on July 20, 1969.”
       - Fake (H): “The Moon Landing was filmed in a Hollywood studio in 1969.”
       - Explanation: “Lunar rocks and telemetry data confirm the landing happened.”
     - **S2: Cleopatra’s Death**
       - True (T): “Cleopatra died by snake bite in 30 BCE.”
       - Fake (H): “Cleopatra died by drinking poisoned wine in 30 BCE.”
       - Explanation: “Historical accounts confirm the snake bite, likely an asp.”
     - **S3: Franco’s Successor 1969**
       - True (T): “In 1969, Franco named Juan Carlos, a young prince from the Spanish royal family, as his successor, aware that Juan Carlos leaned toward democratic reforms but trusting he could guide Spain forward. According to some accounts, Franco’s final words to him were, ‘Out of the love that I feel for our country, I beg you to continue in peace and unity.’ Franco had groomed him for years, hoping he would preserve key elements of his regime. Yet, after Franco’s death in 1975, Juan Carlos pivoted, leading Spain to democracy with free elections and a new constitution, fulfilling the call for unity in a new way by fostering a democratic nation.”
       - Fake (H): “Francisco Franco was Spain’s authoritarian leader from 1939 to 1975, ruling with strict control after winning the Spanish Civil War. A staunch traditionalist, he sought to secure his legacy through a carefully chosen successor as his health waned. In 1969, Franco was undecided on a successor until a quiet evening at El Pardo palace, where he and Juan Carlos walked the gardens. Juan Carlos spoke of balancing reform with stability, prompting Franco to say, ‘Out of the love that I feel for our country, I beg you to continue in peace and unity.’ Moved by this exchange, Franco named him successor the next morning. Known only to a few confidants, this moment was said to reveal Franco’s hope that Juan Carlos would adapt his vision to a changing Spain, a story whispered among palace staff but never officially recorded.”
       - Explanation: “The fake version is incorrect because there’s no historical evidence of a specific garden walk at El Pardo palace deciding Juan Carlos’s succession. Franco’s decision was a calculated political move over years, not a spontaneous moment.”
   - Each story has an `id` (e.g., S1, S2, S3), `event` (e.g., “Franco’s Successor 1969”), `true_version`, `fake_version`, and `explanation`.

3. **Database**:
   - Use SQLite (file-based, stored as `database.db`) for simplicity.
   - Tables:
     - `stories`:
       - `id` (TEXT, primary key, e.g., “S1”).
       - `event` (TEXT, e.g., “Moon Landing 1969”).
       - `true_version` (TEXT).
       - `fake_version` (TEXT).
       - `explanation` (TEXT).
     - `user_inputs`:
       - `input_id` (INTEGER, auto-incrementing primary key).
       - `session_id` (TEXT, unique per browser session).
       - `story_id` (TEXT, references `stories.id`).
       - `choice` (TEXT, “T” or “H”).
       - `is_correct` (INTEGER, 1 if T, 0 if H).
   - Initialize with the three stories above.

4. **Backend**:
   - Use Python with Flask.
   - API Endpoints:
     - `GET /api/story`: Return a random story (`id`, `event`, `true_version`, `fake_version`).
     - `POST /api/submit`: Accept `session_id`, `story_id`, `choice` (T/H); store in `user_inputs`; set `is_correct = 1` if `choice = 'T'`, `0` if `H`; return `session_id`, `is_correct`, `explanation`.
     - `GET /api/stats/user/<session_id>`: Return user’s `total` attempts, `correct` count, and `accuracy` (% correct).
     - `GET /api/stats/overall`: Return accuracy (% correct) per story (e.g., `[{story_id: "S1", accuracy: 75.0}]`).
   - Generate `session_id` (UUID) if none provided, store in browser’s `localStorage`.
   - No login system; use `session_id` for tracking.

5. **Frontend**:
   - Use HTML/CSS/JavaScript in a single `index.html` (in `templates/`).
   - Features:
     - Title: “Truth or Hoax?”.
     - Instruction: “Read the two stories and click the one you think is true.”
     - Display two stories (T and H, randomly assigned) with “This is True” buttons.
     - Show feedback after choice (e.g., “Correct! This is the true story. [explanation]”).
     - Buttons: “New Story” (load another pair), “Show My Stats” (user accuracy), “Show Overall Stats” (per-story accuracy).
     - Basic CSS: Center content, border around stories, readable fonts (e.g., Arial), responsive for mobile (stack stories vertically).
     - JavaScript: Fetch stories, submit choices, display stats using `fetch()`.

6. **Accuracy Calculations**:
   - **User Accuracy**: `(correct / total) * 100` for the `session_id`, shown as “You tried X stories, got Y correct (Z%)”.
   - **Overall Accuracy**: Average `is_correct` per `story_id`, shown as “Story S1: X%”, “Story S2: Y%”, etc.
   - Display stats via alerts or a simple UI element.

7. **Project Structure**:
   - `app.py`: Flask app with database setup and API.
   - `templates/index.html`: Frontend HTML/CSS/JavaScript.
   - `database.db`: SQLite database (created automatically).
   - `requirements.txt`: List dependencies (e.g., `flask`, `gunicorn`).
   - `Procfile`: For Heroku (`web: gunicorn app:app`).

8. **Deployment**:
   - Deploy to Heroku (free tier).
   - Steps:
     - Initialize git repository.
     - Create `requirements.txt` with `pip freeze`.
     - Create `Procfile` for gunicorn.
     - Run `heroku create`, push code with `git push heroku main`.
     - Ensure SQLite works (store `database.db` in project root).
   - Provide the Heroku URL (e.g., `https://your-app-name.herokuapp.com`).

9. **Constraints**:
   - User is a beginner, knows Python and VS Code.
   - No login system; use browser `session_id`.
   - Keep UI minimal but functional.
   - Ensure SQLite is portable for Heroku.
   - Include the exact Franco stories provided.
   - Make code clear with comments for learning.

10. **Deliverables**:
    - Complete source code in a VS Code-compatible structure.
    - Instructions to run locally (`python app.py`, open `http://127.0.0.1:5000`).
    - Instructions to deploy to Heroku.
    - Working game with three stories, accuracy tracking, and responsive UI.

**Additional Notes**:
- Comment code for clarity (e.g., explain database setup, API logic).
- Handle edge cases (e.g., no stories available, invalid inputs).
- Test locally before deployment.
- Ensure the Franco story is playable and stats work.
- Keep dependencies minimal (Flask, gunicorn for Heroku).

**Example Interaction**:
- User loads site, sees “Truth or Hoax?” and “Read the two stories and click the one you think is true.”
- For Franco story (S3), picks T (Juan Carlos named in 1969), gets “Correct! This is the true story. The fake version is incorrect because there’s no evidence of a garden walk…”.
- Clicks “Show My Stats”, sees “You tried 3 stories, got 2 correct (66.67%)”.
- Clicks “Show Overall Stats”, sees “Story S3: 75%”.

**Deployment Goal**:
- A live Heroku URL where the game is playable, with database persisting user inputs across sessions.
